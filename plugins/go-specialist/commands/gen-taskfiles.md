# Generate Taskfiles Command

Instantly create complete Task automation setup for Go projects including main Taskfile, development tasks, and pre-commit hooks configuration. This command provides a comprehensive task runner setup with linting, testing, building, and release automation.

## Why This Command Exists

**Problem**: Setting up task automation for Go projects requires manually creating Taskfile configuration, understanding Task syntax, and configuring pre-commit hooks - a 20-30 minute process.

**Solution**: One-command generation that:
- Creates complete Taskfile setup (3 files, 105 lines total)
- Includes tasks: lint, test, build, snapshot, release, image, doc
- Configures pre-commit hooks for automated quality checks
- Integrates with golangci-lint and goreleaser
- Provides development-specific tasks (pre-commit installation)

## Content Generation Rules

**CRITICAL**: When generating any content (taskfiles, commit messages, or any other output):
- **NEVER** add Claude Code attribution or co-authorship
- **NEVER** include phrases like "Generated by Claude", "Created with Claude Code", or similar
- **NEVER** add "Co-Authored-By: Claude" in commit messages or documentation
- All generated content should appear as if written by a human developer
- Focus on technical accuracy and providing working templates

## Process (6 Phases)

### Phase 1: Prerequisites Validation

**Critical checks before proceeding:**

1. **Validate Git repository:**
   ```bash
   git rev-parse --git-dir
   ```
   - If fails: Exit with error "Not a git repository. Run 'git init' first."

2. **Validate Go project:**
   ```bash
   test -f go.mod
   ```
   - If fails: Exit with error "go.mod not found. This command is for Go projects only."

3. **Check existing taskfiles:**
   ```bash
   test -f Taskfile.yml
   test -f Taskfile_dev.yml
   test -f .pre-commit-config.yaml
   ```
   - Store which files exist: `existingFiles = []`
   - If any exist: Store `hasExistingFiles = true`

4. **Verify command assets directory:**
   - Use Read tool to verify `${CLAUDE_PLUGIN_ROOT}/commands/assets/taskfiles/Taskfile.yml` exists
   - If fails: Exit with error "Command assets directory not found at ${CLAUDE_PLUGIN_ROOT}/commands/assets/taskfiles/. Plugin may be corrupted. Reinstall the go-specialist plugin."

5. **Check for supporting configs (non-blocking):**
   ```bash
   test -f .golangci.yml
   test -f .goreleaser.yml
   ```
   - If .golangci.yml missing: Store `linterMissing = true` (warn in Phase 6)
   - If .goreleaser.yml missing: Store `goreleaserMissing = true` (warn in Phase 6)

### Phase 2: Argument Parsing

Parse command arguments from `$argument` string:

**Supported flags:**
- `--force`: Skip confirmation for existing files
- `--dry-run`: Show preview without writing files

**Parsing logic:**
```
arguments = split($argument by space)
for each arg in arguments:
  if arg == "--force": set forceMode = true
  if arg == "--dry-run": set dryRunMode = true
```

### Phase 3: Template Reading

**Read all template files** from command assets directory:

**Files to read:**

1. **Taskfile.yml** (66 lines)
   - Path: `${CLAUDE_PLUGIN_ROOT}/commands/assets/taskfiles/Taskfile.yml`
   - Target: `Taskfile.yml` (project root)

2. **Taskfile_dev.yml** (14 lines)
   - Path: `${CLAUDE_PLUGIN_ROOT}/commands/assets/taskfiles/Taskfile_dev.yml`
   - Target: `Taskfile_dev.yml` (project root)

3. **.pre-commit-config.yaml** (25 lines)
   - Path: `${CLAUDE_PLUGIN_ROOT}/commands/assets/taskfiles/.pre-commit-config.yaml`
   - Target: `.pre-commit-config.yaml` (project root)

**Template handling:**
- Store content in memory for Phase 3.5 substitution
- All substitutions handled in Phase 3.5 (not inline)

**Error handling:**
- If any Read fails: Exit with error "Failed to read template file: {filename}. Plugin may be corrupted. Ensure ${CLAUDE_PLUGIN_ROOT}/commands/assets/taskfiles/ directory exists."

### Phase 3.5: Auto-Detection & Substitution

**Detection Function Library:** Use the shared detection functions from `${CLAUDE_PLUGIN_ROOT}/commands/assets/detection-functions.md`. Read that file to get all bash detection functions (`detect_project_name`, `detect_git_owner`, `detect_main_dir`, `detect_registry`, `substitute_placeholders`).

**Detection Workflow:**

1. **Detect all values:**
   ```bash
   PROJECT_NAME=$(detect_project_name)
   OWNER=$(detect_git_owner)
   REGISTRY=$(detect_registry)
   MAIN_DIR=$(detect_main_dir)
   ```

2. **Handle missing critical values:**
   - If `PROJECT_NAME` is empty: Use "app" as default
   - If `OWNER` is empty: Preserve `[OWNER]` placeholder
   - Non-critical values have sensible defaults

3. **Substitute placeholders in Taskfile.yml template:**
   ```bash
   taskfile_template=$(substitute_placeholders "$taskfile_template" \
       "$PROJECT_NAME" "$OWNER" "$REGISTRY" "$MAIN_DIR")
   ```

   Note: Taskfile_dev.yml and .pre-commit-config.yaml have no placeholders

### Phase 5: File Creation

**File writing sequence:**

Write files in order:

1. `Taskfile.yml`

2. `Taskfile_dev.yml`

3. `.pre-commit-config.yaml`
   - Write as-is to project root

**For each file:**
- Use Write tool with full file path
- Write content from Phase 3 templates
- Track success/failure status
- Continue with remaining files if one fails (partial success acceptable)

**Error handling:**
- If individual file write fails: Log error, continue with next file
- If all writes fail: Report failure with details
- If partial success: Report which files succeeded/failed

**Track results:**
```
successfulFiles = []
failedFiles = []

for each file:
  try write:
    if success: add to successfulFiles
    if failure: add to failedFiles
```

## Command Arguments

Arguments are parsed from the `argument` string parameter:

- **`--force`**: Skip confirmation prompt and overwrite existing files
  - Use when: Upgrading existing task configuration
  - Behavior: Overwrites files directly without confirmation

- **`--dry-run`**: Preview files without writing to disk
  - Shows: File list, line counts, preview of content
  - Writes: Nothing
  - Use when: Reviewing what will be created before committing

**Parsing example:**
```
argument = "--force"
Split by space, check for each flag:
- forceMode = true
- dryRunMode = false
```

## Task Descriptions

### Main Tasks (Taskfile.yml)

**default**: List all available tasks
- Dependencies: Checks for pre-commit hooks installed
- Command: `task -a`

**check-before-commit**: Run complete quality check suite
- Dependencies: test, snapshot, lint (in order)
- Use: Before pushing code or creating PR

**lint**: Run golangci-lint
- Command: `golangci-lint run`
- Requirements: .golangci.yml file

**test**: Run unit tests with race detection
- Command: `go test -count=2 -race ./...`
- Runs tests twice to catch flaky tests
- Race detector finds concurrency issues

**build**: Build project binaries
- Command: CGO-disabled builds for static binaries
- Customize: Update for your binary names

**image**: Build and push Docker image
- Dependencies: build task
- Customize: Update registry and image name

**doc**: Start godoc documentation server
- Command: `godoc -http=:6060`
- Access: http://localhost:6060

**snapshot**: Create snapshot release
- Command: `goreleaser --clean --snapshot`
- Requirements: .goreleaser.yml file
- Use: Test release process without publishing

**release**: Create production release
- Command: `goreleaser --clean`
- Requirements: .goreleaser.yml file, git tag
- Use: On tagged commits to publish release

### Development Tasks (Taskfile_dev.yml)

**dev:install-pre-commit**: Install pre-commit hooks
- Command: `pre-commit install`
- Run once: After cloning repository

**dev:pre-commit**: Run all pre-commit hooks manually
- Command: `pre-commit run --all-files`
- Use: Test hooks before committing

## Pre-commit Hooks Configuration

**.pre-commit-config.yaml** configures automatic quality checks:

**Standard hooks** (from pre-commit/pre-commit-hooks):
- `check-yaml`: Validate YAML syntax (enabled)
- `check-added-large-files`: Prevent commits of large files (enabled)
- `trailing-whitespace`: Remove trailing whitespace (commented, can enable)
- `end-of-file-fixer`: Ensure files end with newline (commented, can enable)

**Local hooks** (run task commands):
- `unit-tests`: Run `task test` before commit
- `lint`: Run `task lint` before commit
- `build`: Run `task snapshot` before commit

All local hooks use `pass_filenames: false` to run full checks, not per-file.

## Limitations

**This command does NOT:**
- Generate .golangci.yml (use `/linter` command)
- Generate .goreleaser.yml (use `/goreleaser` command)
- Install Task runner or pre-commit tools (manual installation required)
- Modify existing taskfiles intelligently (use `--force` to overwrite)
- Install pre-commit hooks automatically (run `task dev:install-pre-commit`)
- Customize binary names automatically (detects from go.mod, may need manual adjustment)

**Scope boundaries:**
- Only handles task and pre-commit configuration files
- Templates are opinionated (aligned with gitlab-backup project patterns)
- Assumes standard Go project structure
- Requires manual customization for multi-binary projects
- Docker image task requires customization for your registry

## Task vs Make

**Why Task instead of Make?**
- YAML syntax (easier than Makefile)
- Cross-platform (Windows, macOS, Linux)
- Built-in parallel execution
- Better dependency management
- Includes from other taskfiles
- Modern tool designed for development workflows

**Migration from Make:**
If you have a Makefile, you can:
1. Keep both (Task can call make targets)
2. Gradually migrate targets to Taskfile.yml
3. Use Task as the primary interface, delegate to make when needed
