# Generate golangci-lint Configuration Command

Instantly create a professional `.golangci.yml` configuration for Go projects with comprehensive linting coverage. This command converts the multi-step `linter` skill into a one-command solution.

## Why This Command Exists

**Problem**: Setting up golangci-lint requires manually creating YAML configuration, understanding 90+ linter options, and balancing strict quality checks with pragmatic development - a 15-30 minute process prone to over-configuration.

**Solution**: One-command generation that:
- Creates battle-tested `.golangci.yml` (224 lines)
- Enables all linters by default with sensible exclusions
- Balances code quality with development velocity
- Works out-of-the-box for most Go projects
- Includes inline documentation for customization

## Content Generation Rules

**CRITICAL**: When generating any content (configuration files, commit messages, or any other output):
- **NEVER** add Claude Code attribution or co-authorship
- **NEVER** include phrases like "Generated by Claude", "Created with Claude Code", or similar
- **NEVER** add "Co-Authored-By: Claude" in commit messages or documentation
- All generated content should appear as if written by a human developer
- Focus on technical accuracy and providing working templates

## Process (6 Phases)

### Phase 1: Prerequisites Validation

**Critical checks before proceeding:**

1. **Validate Git repository:**
   ```bash
   git rev-parse --git-dir
   ```
   - If fails: Exit with error "Not a git repository. Run 'git init' first."

2. **Validate Go project:**
   ```bash
   test -f go.mod
   ```
   - If fails: Exit with error "go.mod not found. This command is for Go projects only."

3. **Check existing configuration:**
   ```bash
   test -f .golangci.yml || test -f .golangci.yaml
   ```
   - If either exists: Store `existingConfig = true` and `existingFile = <filename>`
   - If not exists: Store `existingConfig = false`

4. **Check for golangci-lint binary** (non-blocking):
   ```bash
   command -v golangci-lint
   ```
   - If not found: Store `linterNotInstalled = true` (will show warning later)
   - If found: Check version with `golangci-lint --version` and store for report

5. **Verify command assets directory:**
   - Use Read tool to verify `${CLAUDE_PLUGIN_ROOT}/commands/assets/linter/.golangci.yml` exists
   - If fails: Exit with error "Command assets directory not found at ${CLAUDE_PLUGIN_ROOT}/commands/assets/linter/. Plugin may be corrupted. Reinstall the go-specialist plugin."

### Phase 2: Argument Parsing

Parse command arguments from `$argument` string:

**Supported flags:**
- `--force`: Skip confirmation if .golangci.yml already exists
- `--dry-run`: Show preview without writing file

**Parsing logic:**
```
arguments = split($argument by space)
for each arg in arguments:
  if arg == "--force": set forceMode = true
  if arg == "--dry-run": set dryRunMode = true
```

**File Selection:**
- Single file generation: `.golangci.yml` (224 lines)

### Phase 3: Template Reading

**Read template file** from command assets directory:

**Template location:**
- Path: `${CLAUDE_PLUGIN_ROOT}/commands/assets/linter/.golangci.yml`
- Expected size: ~6.5 KB (224 lines)
- Format: YAML with inline comments

**Reading logic:**
```
Use Read tool to read: ${CLAUDE_PLUGIN_ROOT}/commands/assets/linter/.golangci.yml
Store content as: templateContent
Verify: content is not empty and starts with "# golangci-lint"
```

**Error handling:**
- If file not found: Exit with error "Template file not found at ${CLAUDE_PLUGIN_ROOT}/commands/assets/linter/.golangci.yml"
- If file empty: Exit with error "Template file is empty or corrupted"

**No variable substitution required** - the template is copied as-is.

### Phase 4: User Confirmation

**Build preview message:**

```

Files to be created:
  .golangci.yml (224 lines, ~6.5 KB)

  • Enables all linters by default (90+)
  • Disables overly strict linters (goerr113, wrapcheck, etc.)
  • 5-minute timeout for large projects
  • Fast mode with caching enabled
  • Parallel execution across all CPUs
  • Project-agnostic settings

Excluded linters (too opinionated):
  • depguard, exhaustruct, gci, gochecknoglobals
  • godox, goerr113, inamedparam, maintidx
  • nonamedreturns, testpackage, wrapcheck, wsl

Integration ready:
  ✓ GitHub Actions (via gen-github-dir)
  ✓ GitLab CI (via gen-gitlab-ci)
  ✓ Task runner (via gen-taskfiles)
  ✓ Pre-commit hooks
  ✓ VS Code, GoLand, Vim/Neovim
```


**Confirmation logic:**

If `dryRunMode = true`:
```

```

If `forceMode = true`:
```

```
Proceed to Phase 5

If `existingConfig = true` and `forceMode = false`:
```

  Question: "Overwrite existing .golangci.yml configuration?"
  Options:
    - "Yes, overwrite configuration" (proceed to Phase 5)
    - "No, cancel operation" (exit with message "Operation cancelled by user")
```

If `existingConfig = false`:
```

  Question: "Create .golangci.yml in project root?"
  Options:
    - "Yes, create configuration" (proceed to Phase 5)
    - "No, cancel operation" (exit with message "Operation cancelled by user")
```

### Phase 5: File Creation

**Write configuration file:**

Use Write tool to create:
```

```

**Writing logic:**
```

  file_path: ./.golangci.yml (absolute path to project root)
  content: templateContent (no modifications, copy as-is)
```

**Error handling:**
- If write fails: Report error to user with file path and error message
- If write succeeds: Proceed to Phase 6

## Best Practices

**Configuration maintenance:**
- Review and update `.golangci.yml` when upgrading golangci-lint versions
- Adjust linter selection based on team preferences
- Document any custom exclusions in comments
- Test configuration changes with `--fast` flag first

**CI/CD integration:**
- Always run linter in CI before tests
- Use `--timeout=5m` for large projects
- Cache golangci-lint binary and build cache
- Block PRs with linting failures

**Local development:**
- Run linter before committing: `golangci-lint run ./...`
- Use auto-fix when possible: `golangci-lint run --fix ./...`
- Configure editor integration for real-time feedback
- Set up pre-commit hooks for automatic checks

**Team adoption:**
- Gradually enable stricter linters
- Document project-specific conventions
- Review linter output in code reviews
- Share configuration explanations with team

## Limitations & Caveats

**Golangci-lint installation:**
- This command only creates configuration
- Binary must be installed separately
- Requires Go 1.21+ for full linter compatibility

**Configuration style:**
- Opinionated defaults (pragmatic over perfect)
- May need customization for specific projects
- Some linters may conflict with project conventions

**Performance considerations:**
- Initial run can be slow on large codebases
- Enable caching for faster subsequent runs
- Use `--fast` flag for quick checks during development

**CI/CD requirements:**
- Workflows expect golangci-lint v2.4.0+
- Docker images may need golangci-lint installation
- GitHub Actions uses dedicated golangci-lint action

## Error Recovery

**If linter fails after generation:**
1. Check golangci-lint version: `golangci-lint --version`
2. Validate configuration: `golangci-lint run --help`
3. Test with minimal config: `golangci-lint run --disable-all --enable=govet ./...`
4. Check for syntax errors: `yamllint .golangci.yml`

**If too many issues reported:**
1. Start with auto-fix: `golangci-lint run --fix ./...`
2. Disable problematic linters temporarily
3. Use `--exclude` patterns for known issues
4. Gradually enable stricter linters

**If conflicts with existing tooling:**
1. Review `.golangci.yml` disable list
2. Adjust exclusions for project needs
3. Check for conflicting editor plugins
4. Verify Go version compatibility
